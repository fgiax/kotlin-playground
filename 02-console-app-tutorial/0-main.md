# Gradle init (Application)

> Aplication это в смысле упаковка, 

Не совсем.

Application в Gradle init — это не “упаковка”, а шаблон проекта для запускаемого приложения (executable program).

Разница важная:

📦 Application = “у этого проекта есть main() и его надо запускать”

Он говорит Gradle:

у проекта есть точка входа (main)

его нужно уметь запускать

его нужно уметь распространять как программу


Поэтому добавляются:

run → запуск из исходников

installDist → готовая папка с программой

distZip/distTar → архив приложения



---

📦 Но это НЕ “упаковка в jar”

Gradle не считает, что:

> application = сделать один jar-файл



По умолчанию:

jar → просто архив классов (как библиотека)

application → способ запускать и раскладывать программу по папкам



---

🧠 Простая модель

library project  → jar (используется другими)
application      → run + installDist + distZip


---

💡 Почему тебя это путает

Потому что в Java-мире часто думают:

> приложение = один jar



Но в Gradle:

> приложение = папка + скрипты + зависимости




---

✔️ Итог

Фраза:

> “Application это упаковка”



❌ не совсем верна

Правильнее:

> Application — это шаблон Gradle для проекта с main(), который предназначен для запуска и распространения как программы (не обязательно одним jar-файлом)

## Gradle не считает, что "приложение = исполняемый JAR".

Когда ты выбрал Application, Gradle сделал следующее:

- добавил плагин application;
- создал задачу run;
- попросил указать главный класс (mainClass);
- добавил задачи installDist, distZip, distTar.


Но задача jar не изменилась. Она по-прежнему собирает обычный JAR с классами проекта.

Идея Gradle такая:

Приложение
├── run          ← запуск
├── installDist  ← готовая папка для запуска
├── distZip      ← архив приложения
└── jar          ← просто JAR с классами

То есть jar — это не "собрать приложение", а "упаковать классы".

Почему так сделано?

Потому что приложение может зависеть от десятков библиотек.

Если сделать один исполняемый JAR автоматически, возникнет вопрос: что делать с зависимостями? Копировать внутрь? Оставить рядом? Искать их в Maven? У разных проектов разные требования.

Поэтому Gradle по умолчанию распространяет приложение как каталог:

build/install/app/
├── bin/
│   └── app      ← скрипт запуска
└── lib/
    ├── app.jar
    ├── kotlin-stdlib.jar
    └── ...

Этот каталог можно запускать сразу, и все зависимости уже лежат рядом.

Если же нужен один файл, то это уже отдельная задача. Для неё обычно используют плагин Shadow, который собирает fat JAR (app-all.jar) со всеми зависимостями.

Именно поэтому многие новички удивляются: выбрал "Application", а jar всё равно похож на библиотеку. Это нормальное поведение Gradle — он разделяет понятия "архив с классами" и "готовое к распространению приложение".
